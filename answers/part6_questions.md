# Часть 6: Контрольные вопросы

## 1) В каком порядке выполняются конструкторы при создании объекта подкласса?
Ваш ответ: Сначала всегда вызывается конструктор родительского класса, затем — конструктор подкласса.
Пример: если есть class Dog extends Animal, то при создании new Dog() сначала отработает конструктор Animal(), потом Dog().
Типичная ошибка: забыть, что у родителя нет конструктора без параметров. Если в Animal есть только Animal(String name),
то в Dog нужно явно вызвать super(name), иначе компилятор выдаст ошибку.

## 2) Можно ли переопределить `static` метод? Что такое method hiding?
Ваш ответ: Нельзя, статические методы нельзя переопределить. Если в подклассе написать статический метод с такой же
сигнатурой, как в родителе, это называется method hiding (сокрытие метода).
Определение: Сокрытие метода — это когда подкласс объявляет свой статический метод с тем же именем и параметрами,
что и в родителе. При этом метод родителя не переопределяется, а просто "прячется" за новым.
Типичная ошибка новичка: Думать, что статические методы работают как обычные и переопределяются. На самом деле они
привязаны к классу, и полиморфизм для них не работает. Если поставить аннотацию @Override над статическим методом,
компилятор вообще выдаст ошибку.

## 3) Что произойдет, если в `switch` нет `default` и ни один `case` не совпал?
Ваш ответ: Ничего. Программа просто пропустит весь switch и продолжит выполнение дальше.
Пример: если int x = 5, а case 1, case 2, то код внутри switch не выполнится.
Типичная ошибка: надеяться, что switch сам выбросит ошибку или вызовет default. Если случай не предусмотрен,
лучше явно написать default с логированием.

## 4) Почему в `for-each` нельзя удалять элементы коллекции? Как правильно?
Ваш ответ:Потому что for-each внутри использует итератор, а при удалении элемента напрямую из коллекции итератор
"портится" и выбрасывает ConcurrentModificationException.
Пример ошибки: for (String s : list) { list.remove(s); }
Правильно: использовать обычный итератор и его метод remove(), или идти по коллекции с конца обычным for.
Типичная ошибка: пытаться удалять элементы через list.remove() внутри for-each — это почти всегда ведёт к исключению.

## 5) Чем отличается `break` с меткой от обычного `break`?
Ваш ответ: Обычный break выходит только из самого внутреннего цикла. break с меткой выходит из всех
циклов до той метки, которую указали.
Пример: outer: for(...) { for(...) { break outer; } } — выйдет сразу из обоих циклов.
Типичная ошибка: пытаться выйти из внешнего цикла без метки, используя обычный break. Это не сработает — выйдет только
из внутреннего.

## 6) Что означает `@Override`?
Ваш ответ: Это аннотация, которая говорит компилятору: "я уверен, что переопределяю метод родителя". Если метода в
родителе нет или сигнатура не совпадает, компилятор выдаст ошибку.
Пример: @Override public void makeSound() — если в родителе makeSound() с другим параметром, будет ошибка.
Типичная ошибка: опечатка в имени метода (makSound) без @Override создаст новый метод, а не переопределение,
и полиморфизм не сработает.

## 7) Какой модификатор доступа выбрать для видимости в подклассах?
Ваш ответ: protected. Поля и методы с protected видны в подклассах (даже в других пакетах) и внутри своего пакета.
Пример: protected int age; — подкласс Dog может напрямую читать и менять age.
Типичная ошибка: сделать поле private, а потом удивиться, что подкласс не видит его. Или сделать public, нарушив инкапсуляцию.

## 8) Разница между `final` классом и `sealed` классом?
Ваш ответ: final класс вообще нельзя наследовать. sealed класс можно наследовать, но только строго определённым классам,
которые перечислены в permits.
Пример: final class String — никто не может создать подкласс.
sealed class Animal permits Dog, Cat — только Dog и Cat могут быть наследниками.
Типичная ошибка: думать, что sealed класс нельзя наследовать вообще — это не так, он разрешает, но только избранным.

## 9) Можно ли иметь несколько конструкторов?
Ваш ответ: Да, это называется перегрузкой конструкторов. Они должны отличаться параметрами (количество, тип, порядок).
Пример: public Dog() и public Dog(String name). Один конструктор может вызывать другой через this().
Типичная ошибка: пытаться сделать два конструктора с одинаковыми параметрами — это ошибка компиляции. Или забыть,
что this() должен быть первой строкой.

## 10) Что такое ковариантный тип возвращаемого значения?
Ваш ответ: Это когда переопределённый метод может возвращать не тот же тип, а его подкласс.
Пример: родительский метод возвращает Animal, а в подклассе можно вернуть Dog (потому что Dog — это Animal).
Типичная ошибка: думать, что тип возврата должен совпадать точь-в-точь. На самом деле можно
сужать (возвращать более конкретный тип). Расширять (возвращать родителя) нельзя.
